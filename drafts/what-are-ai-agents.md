# What are AI agents?

Статус: первый черновик.

Дата: 2026-05-16

Тип: собственный синтез с опорой на карточки источников проекта.

## 1. Главная идея

AI [agents](../notes/concepts/agent.md) удобнее всего понимать не как новый вид нейросети, а как новый способ собрать систему вокруг модели. В обычном LLM-приложении модель чаще всего получает запрос и возвращает ответ. В агентской системе модель становится частью цикла действия: она получает цель, выбирает следующий шаг, использует инструмент или взаимодействует со средой, получает и анализирует результат действия (observation) и продолжает работу.

Иначе говоря, сдвиг происходит от "модель отвечает" к "система выполняет задачу". Это не отменяет важность самой модели, но переносит центр внимания на связку: модель, контекст, инструменты, состояние, правила контроля и оценка результата.

Рабочее определение:

> AI agent — это система, в которой модель не только генерирует текст, но и участвует в выборе действий, использует инструменты, получает и анализирует результаты действий (observations), обновляет состояние задачи и продолжает работу до достижения цели или остановки.

Это определение намеренно говорит о системе, а не только о модели. Агентность возникает не из одного prompt, а из всей архитектуры вокруг модели.

## 2. От chatbot к agentic system

Путаница начинается с того, что многие агентские системы имеют чатовый интерфейс. Пользователь пишет задачу в текстовом поле, система отвечает текстом, и внешне это похоже на chatbot. Но интерфейс не определяет архитектуру.

Chatbot — это способ общения. Он может быть простым ответчиком, а может быть оболочкой вокруг сложного агента. Если система только формулирует ответ на реплику пользователя, ее лучше называть chatbot или LLM assistant. Если она может сама выбрать инструмент, выполнить действие, прочитать результат и продолжить работу, это уже agentic system.

Вторая важная граница — между agent и workflow. [Workflow](../notes/concepts/workflow.md) заранее задает маршрут: шаги, условия, переходы, вызовы API. LLM может быть одним из шагов внутри такого процесса. Agent отличается тем, что часть маршрута остается открытой: модель участвует в выборе следующего действия на основе цели и observations (наблюдений результатов).

На практике это не бинарное различие. Надежные прикладные системы часто гибридные. Workflow задает рамки, права, этапы и остановку. Agent выбирает действия внутри этих рамок. Такой гибрид особенно важен там, где есть риск: финансовые операции, изменение данных, отправка сообщений, деплой кода, работа с приватной информацией.

Хорошее практическое правило: если модель только заполняет один шаг в заранее заданном процессе, это LLM workflow. Если модель выбирает следующий шаг и взаимодействует со средой, это agentic system.

## 3. Минимальная архитектура агента

Упрощенный агентский цикл можно записать так:

```text
goal -> context -> model decision -> tool/action -> observation -> updated state -> next decision
```

`Observation` — общепринятый термин агентского цикла. В этом тексте он означает получение и анализ результата действия: ответ API, вывод инструмента, изменение среды, найденный документ или другой сигнал, который влияет на следующий шаг.

В этом цикле есть несколько обязательных компонентов.

Model — LLM или мультимодальная модель, которая понимает задачу, интерпретирует контекст и помогает выбрать действие.

Tools — внешние возможности: функции, API, поиск, браузер, файловая система, база данных, кодовая среда, корпоративные сервисы.

State или [memory](../notes/concepts/memory.md) — то, что система сохраняет между шагами: цель, план, результаты инструментов, промежуточные выводы, ошибки, ограничения.

Orchestration — слой управления: какие инструменты доступны, как выполняются вызовы, как возвращаются результаты, когда нужен human approval, когда задача должна остановиться.

Environment — среда, в которой агент действует и получает обратную связь: веб-страница, кодовая база, операционная система, CRM, таблица, симуляция, пользовательский документ.

Evaluation — способ понять, получилось ли выполнить задачу и были ли действия разумными, безопасными и воспроизводимыми.

Главный урок здесь простой: агент — это не "LLM с длинным prompt". Это управляемая система действий.

## 4. Tool use как поворотный пункт

[Tool use](../notes/concepts/tool-use.md) — одна из причин, почему агентские системы стали практичными. Без инструментов модель ограничена тем, что уже находится в ее параметрах и текущем контексте. С инструментами она может искать актуальную информацию, читать файлы, делать вычисления, вызывать API, запускать код, управлять браузером или взаимодействовать с приложениями.

Важно различать function calling и tool use. Function calling — конкретный механизм структурированного вызова функции. Tool use — более широкий слой: custom functions, built-in tools, browser use, computer use, code execution, file search, remote [MCP](../notes/concepts/protocols-and-interoperability.md) servers.

В современных API tool use становится частью базовой платформы. Например, OpenAI Responses API описывает tools как способ расширить возможности модели через built-in tools, custom function calls и remote MCP servers. OpenAI Agents SDK, LangGraph, AutoGen и CrewAI по-разному строят orchestration вокруг того же фундаментального вопроса: как дать модели действовать через инструменты и при этом сохранить контроль.

Исследовательски важными предшественниками здесь являются ReAct и Toolformer. ReAct показывает паттерн reason -> act -> observe, где рассуждение чередуется с действием и observation — получением и анализом результата действия. Toolformer исследует, как модель может научиться решать, когда и как вызывать внешние API. Эти работы не покрывают всю современную production-архитектуру агентов, но помогают понять, почему tool use стал центральным элементом.

## 5. MCP и инфраструктура вокруг агентов

Когда у агента появляется много инструментов, возникает инфраструктурный вопрос: как подключать эти инструменты так, чтобы каждое приложение не изобретало собственный способ интеграции?

MCP (Model Context Protocol) важен именно здесь. Его можно понимать как попытку стандартизировать слой между AI-приложением и внешними возможностями: tools, resources, prompts и контекст. Для агентских систем это не просто удобный API. Это часть среды, в которой агент может действовать.

Если tool use отвечает на вопрос "что агент может сделать?", то MCP отвечает на вопрос "как разные приложения и внешние системы могут предоставить агенту эти возможности единообразно?".

При этом MCP сразу поднимает вопросы безопасности: какие серверы можно подключать, кто доверяет описанию tool schema, какие разрешения нужны, как журналировать вызовы, как ограничивать опасные действия. Чем удобнее подключение инструментов, тем важнее модель прав и наблюдаемости.

## 6. Memory: не одно понятие, а несколько

В агентских системах слово "memory" часто вводит в заблуждение. Оно может означать очень разные вещи.

[Context window](../notes/concepts/context-window.md) — информация, прямо переданная модели в текущем запросе.

[Session state](../notes/concepts/session-state.md) — состояние текущей задачи между шагами: цель, план, результаты инструментов, ошибки, промежуточные решения.

[Retrieval](../notes/concepts/retrieval-rag.md) или RAG — поиск релевантных фрагментов во внешнем хранилище и добавление их в контекст.

[Long-term memory](../notes/concepts/long-term-memory.md) — данные, которые сохраняются между сессиями: предпочтения пользователя, история проектов, устойчивые факты.

[Skill library](../notes/concepts/skill-library.md) — сохраненные процедуры или код, которые агент может повторно использовать. Voyager особенно интересен именно этим: агент в Minecraft накапливает библиотеку исполняемых навыков и переиспользует их в новых ситуациях.

[Audit trail](../notes/concepts/audit-trail.md) — история действий и решений, нужная не для "памяти" модели, а для проверки, отладки и доверия.

Практически полезно начинать с явного session state и audit trail. Долговременную память стоит добавлять только тогда, когда понятно, что именно нужно помнить, кто может это увидеть, как это исправить и как проверить, что память не портит результат.

## 7. Planning и контроль

[Planning](../notes/concepts/planning.md) — способность разбить цель на шаги и менять порядок действий по мере появления новых observations (наблюдений результатов). Иногда план существует только внутри следующего решения модели. Иногда система создает явный список шагов. Иногда control flow задан графом, как в graph-based orchestration.

Но план сам по себе не гарантирует надежность. Модель может написать убедительный план, который не связан с реальностью. Поэтому важнее не наличие красивого плана, а связь между планом, действиями, observations (наблюдениями результатов) и критериями остановки.

Хорошая агентская система должна отвечать на несколько вопросов:

- что агент собирается сделать;
- какие инструменты ему разрешены;
- что произойдет при ошибке инструмента;
- когда нужно спросить человека;
- как понять, что задача завершена;
- как восстановить ход работы после сбоя.

Именно поэтому production agents обычно требуют orchestration, guardrails и human control, а не только сильной модели.

## 8. Multi-agent systems

[Multi-agent systems](../notes/concepts/multi-agent.md) используют несколько агентов или ролей: planner, researcher, executor, critic, reviewer. Эта идея интуитивно привлекательна, потому что напоминает команду специалистов.

Она действительно может быть полезна, когда задача естественно делится на разные компетенции, когда нужен независимый review, когда можно параллелить исследование или когда разные роли должны иметь разные инструменты и права.

Но несколько агентов не являются автоматическим улучшением. Они могут повторять работу друг друга, усиливать ошибки, тратить время на координацию или создавать видимость разумного обсуждения без улучшения результата. Поэтому multi-agent подход должен доказывать пользу через качество, скорость, надежность или управляемость.

AutoGen и CrewAI полезны как практические источники для изучения role-based и conversation-based multi-agent систем. LangGraph полезен как контраст: там акцент сильнее на состоянии, графе и контроле потока. Эти подходы можно сравнивать не как "кто лучше", а как разные ответы на один вопрос: где должен находиться контроль — в разговоре агентов, в ролях, в графе или в приложении?

## 9. Evaluation: оценивать нужно не только ответ

Обычный LLM можно оценивать по качеству ответа. Агентскую систему так оценивать недостаточно, потому что важен не только результат, но и траектория.

Агент мог дать правильный финальный ответ, но сделать лишние tool calls, использовать небезопасный путь, проигнорировать ограничения или случайно зависеть от нестабильного поведения среды. И наоборот: агент мог разумно действовать, но остановиться из-за недоступного инструмента. Для продукта эти случаи важно различать.

Минимальный набор измерений:

- task success: достигнута ли цель;
- trajectory quality: были ли действия разумными;
- safety: не сделал ли агент опасных или лишних действий;
- robustness: сохраняется ли качество при вариациях входа;
- cost and latency: сколько времени и ресурсов ушло;
- human review burden: сколько вмешательства потребовалось.

Benchmarks вроде AgentBench и SWE-bench важны, потому что оценивают модели в интерактивных или исполняемых средах. SWE-bench особенно показателен для coding agents: модель должна не просто написать код, а изменить реальный репозиторий так, чтобы прошли тесты. Но даже такие benchmarks не покрывают все. Passing tests не равен поддерживаемому продукту, а benchmark environment не равен production.

Для практической агентской системы нужны собственные [evals](../notes/concepts/evals.md): успешные сценарии, ошибки инструментов, ambiguous requests, permission boundaries, cases that require human approval и regression tests на прошлые сбои.

## 10. Риски

Агентские системы рискованнее обычных чат-интерфейсов, потому что они действуют во внешней среде и могут накапливать состояние.

Главные риски:

- tool misuse: неправильный или опасный вызов инструмента;
- over-autonomy: система действует там, где нужен человек;
- context error: агент делает вывод из неполной или нерелевантной памяти;
- cascading failure: ранняя ошибка влияет на следующие шаги;
- prompt injection: внешняя информация пытается изменить поведение агента;
- poor observability: невозможно понять, почему был сделан шаг;
- evaluation gap: demo работает, но production cases не покрыты тестами.

Снижение риска начинается с архитектуры. Нужно ограничивать инструменты по задаче, добавлять approval для опасных действий, вести audit trail, проверять траектории, предпочитать workflow там, где путь известен, и увеличивать автономность только там, где она действительно дает пользу.

## 11. Экосистема

Вокруг AI agents уже сложилось несколько слоев.

[SDK and orchestration](../01-glossary.md#sdk-and-orchestration): OpenAI Agents SDK, LangGraph, AutoGen, CrewAI. Они помогают собирать агентов, управлять инструментами, состоянием, ролями, handoffs и выполнением.

Protocols: MCP и смежные стандарты, которые пытаются нормализовать подключение инструментов и контекста.

Runtime: browser use, computer use, sandbox execution, long-running tasks, background execution.

Observability and evals: tracing, replay, trajectory evaluation, guardrails, regression tests.

Product categories: coding agents, research agents, support agents, data agents, operations agents.

Главный вывод: агентский AI — это не только скачок в моделях. Это появление прикладной инфраструктуры, которая превращает модели в управляемые системы выполнения задач.

## 12. Рабочая гипотеза

AI agents стоит изучать как переход от "интеллект в ответе" к "интеллект в процессе". Модель остается центральной, но настоящая агентность появляется в процессе: цель, действие, observation, состояние, контроль, оценка.

Самый продуктивный вопрос поэтому звучит не "насколько агент автономен?", а "какая степень автономности полезна для этой задачи, чем она ограничена и как мы поймем, что система сработала хорошо?".
