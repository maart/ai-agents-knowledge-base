# Session state

Тип: собственный синтез.

## Основная идея

Session state — это состояние текущей задачи между шагами. В отличие от [context window](context-window.md), оно хранится приложением или [orchestration layer](../../01-glossary.md#sdk-and-orchestration) и может выборочно передаваться модели.

## Что хранить

Для агентской задачи session state обычно включает:

- исходную цель;
- текущий план;
- выполненные действия;
- observations после tool calls;
- промежуточные выводы;
- ошибки и retry history;
- ограничения и approvals;
- статус выполнения.

## Зачем нужен session state

Без session state агент может потерять ход задачи: повторять уже сделанное, забывать ограничения, не учитывать ошибку инструмента или преждевременно завершать работу.

Session state помогает системе быть не просто последовательностью независимых LLM-вызовов, а управляемым многошаговым процессом.

## Отличие от long-term memory

Session state живет внутри одной задачи или сессии. [Long-term memory](long-term-memory.md) сохраняется между сессиями и влияет на будущие задачи.

## Риски

- Хранить слишком много и загрязнить последующие шаги.
- Хранить слишком мало и потерять причинно-следственную связь.
- Смешать факты, гипотезы и ошибки.
- Не отделить внутренние рассуждения от проверяемых observations.

## Практическое правило

Для v1 [agentic system](agent.md) session state важнее долговременной памяти. Сначала нужно научиться надежно сохранять состояние текущей задачи, и только потом добавлять память между задачами.
