# Agent

Тип: собственный синтез.

## Основная идея

Agent — это AI-система, в которой модель участвует в выборе действий, использует инструменты, получает и анализирует результаты действий (observation) и продолжает работу в несколько шагов для достижения цели.

Важно: agent — это не просто "умная модель". Агентность появляется в системе: модель + инструменты + состояние + правила контроля + среда выполнения + оценка результата.

## Что делает систему агентской

У агентской системы обычно есть:

- цель, которая может требовать нескольких шагов;
- контекст задачи;
- модель, которая помогает выбрать следующий шаг;
- инструменты или внешняя среда;
- observations: наблюдения результатов действий или tool calls, которые система получает и анализирует;
- состояние между шагами;
- правила остановки;
- ограничения и проверки.

Минимальный цикл:

```text
goal -> observe -> decide -> act -> observe -> update state -> continue or stop
```

`Observe` и `observation` — общепринятые англоязычные термины. В этом проекте `observation` означает не пассивное "смотреть", а получение и анализ результата действия: ответа API, вывода программы, изменения страницы, найденного документа или другого сигнала среды.

Важно отделять агентность от обычной логики ветвлений. Условие вроде "если ссылка есть, прочитать ресурс; иначе идти дальше" может быть частью [workflow](workflow.md). Агентность начинается там, где модель сама выбирает следующий значимый шаг на основе цели, контекста и observations (наблюдений результатов).

## Чем agent полезен

Agent полезен там, где нельзя заранее полностью описать маршрут:

- нужно исследовать неизвестную среду;
- требуется выбирать инструменты по ситуации;
- задача зависит от промежуточных результатов;
- возможны ошибки, и их нужно исправлять по ходу;
- надо работать с файлами, вебом, кодом, таблицами или внешними системами;
- результат требует нескольких итераций.

Простой пример: "найди причину ошибки в кодовой базе и предложи patch". Нельзя заранее знать, какие файлы надо открыть, какие тесты запустить и где окажется проблема. Агентская система может читать, проверять, исправлять и переоценивать задачу по мере работы.

## Agent и автономность

Autonomy — это степень свободы агента. Она может быть разной:

- low autonomy: агент предлагает следующий шаг, человек подтверждает;
- bounded autonomy: агент сам действует внутри ограниченного набора инструментов;
- high autonomy: агент сам планирует и выполняет длинную задачу с редким human review.

Больше автономности не всегда лучше. Хороший дизайн выбирает минимальную автономность, достаточную для задачи.

## Компоненты агента

Model: понимает задачу, интерпретирует observations (наблюдения результатов), выбирает действия.

Tools: функции, API, поиск, браузер, файловая система, кодовая среда, базы данных.

State: текущая цель, история действий, результаты инструментов, промежуточные выводы.

[Planning](planning.md): явный или неявный выбор следующих шагов.

Orchestration: управление вызовами, правами, ошибками, handoffs и stopping conditions.

Guardrails: ограничения, проверки, approvals и политики безопасности.

[Evals](evals.md): способ понять, удалось ли выполнить задачу и была ли траектория разумной.

## Когда agent лучше workflow

Agent чаще подходит, если:

- путь заранее неизвестен;
- задача требует исследования;
- среда меняется;
- нужно выбирать между разными инструментами;
- важны итерации и исправление ошибок;
- нельзя заранее перечислить все ветки процесса.

## Когда agent не нужен

Agent может быть лишним, если:

- процесс хорошо известен;
- достаточно одного LLM-вызова;
- цена ошибки высокая, а действия легко формализуются;
- нужен строгий compliance;
- важнее воспроизводимость, чем гибкость;
- агентская свобода не улучшает результат.

В таких случаях [workflow](workflow.md) часто будет надежнее.

## Главный риск

Agent увеличивает способность системы действовать, но вместе с этим увеличивает поверхность ошибки. Если агент может делать больше, нужно лучше проектировать права, журналирование, остановку, review и evals.

Практическое правило: агентность должна быть функцией задачи, а не украшением архитектуры.
