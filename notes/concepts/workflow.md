# Workflow

Тип: собственный синтез.

## Основная идея

Workflow — это способ собрать AI-систему как управляемый процесс: с понятными шагами, условиями, входами, выходами, проверками и правилами остановки.

В контексте AI agents workflow не является "менее умным" вариантом. Часто это более надежный и профессиональный способ использовать модели, особенно когда задача повторяемая, ответственная или хорошо описывается заранее.

## Почему workflow полезен

Workflow делает систему:

- понятной: можно объяснить, какие шаги выполняются и почему;
- тестируемой: каждый этап можно проверять отдельно;
- воспроизводимой: одинаковые входы чаще дают предсказуемый маршрут;
- безопасной: опасные действия можно вынести в отдельные этапы с approval;
- наблюдаемой: проще логировать, где возникла ошибка;
- управляемой: легче ограничить инструменты, права и условия остановки.

## Как LLM вписывается в workflow

Модель может быть одним или несколькими шагами внутри процесса:

- классифицировать запрос;
- извлечь структурированные данные;
- выбрать следующий маршрут из заранее заданных вариантов;
- сгенерировать черновик;
- проверить качество результата;
- объяснить ошибку или предложить исправление.

В таком дизайне модель не обязательно управляет всем процессом. Она выполняет конкретную интеллектуальную функцию внутри заранее спроектированной системы.

## Workflow и агентность

Workflow и agent не конкурируют как "старое" и "новое". Это две разные степени свободы.

Workflow подходит, когда путь известен. Agent подходит, когда путь нужно открывать по ходу работы. Сильные системы часто соединяют оба подхода:

- workflow задает рамки, этапы, права и остановку;
- agent выбирает действия внутри ограниченного участка;
- человек подтверждает рискованные переходы;
- evals проверяют не только результат, но и маршрут.

## Условные развилки не равны агентности

Workflow может быть сложным и иметь много ветвлений. Это не делает его агентом автоматически.

Например, в n8n можно построить процесс:

```text
получить запрос -> проверить, есть ли ссылка -> если есть, прочитать ресурс -> иначе идти дальше -> собрать ответ
```

Это все еще workflow, если логика ветвлений заранее задана человеком. Модель может суммировать ресурс, извлекать данные или писать черновик, но она не обязательно выбирает маршрут.

Agentic component появляется там, где модель получает цель, контекст и набор допустимых действий, а затем сама выбирает следующий значимый шаг. Например: читать ли ссылку, искать ли дополнительный источник, достаточно ли данных для ответа, нужно ли запросить уточнение.

Такой дизайн лучше называть agentic workflow: workflow задает рамки, а модель получает ограниченную свободу выбора внутри них.

## Примеры workflow-подхода

- Обработка входящего письма: классификация -> извлечение данных -> поиск клиента -> черновик ответа -> human review.
- Исследовательская заметка: поиск источников -> карточки источников -> синтез -> проверка ссылок -> публикация.
- Работа с документом: загрузка -> извлечение структуры -> редактирование -> проверка форматирования -> экспорт.
- Поддержка клиента: определение категории -> поиск в базе знаний -> предложение ответа -> эскалация при риске.
- Coding task: анализ issue -> поиск файлов -> patch -> tests -> review.

## Когда workflow лучше агента

Workflow часто предпочтительнее, если:

- шаги известны заранее;
- цена ошибки высокая;
- нужны audit trail и compliance;
- важна повторяемость;
- есть четкие business rules;
- задача должна легко тестироваться;
- модель не должна сама решать, какие действия разрешены.

## Ограничения workflow

Workflow может стать слишком жестким, если задача плохо предсказуема. Он может ломаться на нестандартных случаях, требовать много ручного проектирования и плохо работать там, где нужно исследование или адаптация.

Поэтому вопрос не в том, "workflow или agent", а в том, где нужна структура, а где нужна автономность.

## Практическое правило

Начинай с workflow, если можешь описать процесс. Добавляй агентность только там, где заранее заданного маршрута недостаточно.
